### 🔍 測試任務背景（Testing Mission Context）

本次 pair testing 的目標，是在**尚未開發實作前**，協助團隊釐清 Phase 1 登入與身份驗證機制中，可能影響**存取邊界、安全性與角色隔離**的高風險區域。

系統存在兩種登入角色與機制：

* awoo 操作管理員：Google SSO + 白名單
* 品牌方使用者：帳號 + 金鑰（token）

這兩種機制**共用同一個平台與資料入口**，因此任何驗證或狀態錯誤，都可能造成跨品牌資料外洩或權限誤用。

---

### 🧠 我們目前對系統的理解模型（初步）

#### 角色模型

* Internal Admin（awoo）
* External Brand User（品牌方）

#### 身分憑證模型

* Google SSO Email（domain + whitelist）
* Brand Account + Token
* Browser Session（7 日）

#### 狀態模型（簡化）

* 未登入
* 已登入（有效 session）
* session 過期
* token 被更新 / 作廢
* 合約終止（尚未明確定義系統行為）

---

### ⚠️ 初步辨識出的關鍵風險區域

#### 1️⃣ 白名單與 Google SSO 的邊界風險

**我們在擔心什麼：**

* domain 判斷是否只看 email domain？
* 是否存在 alias / secondary domain 被誤接受的情境？
* 白名單資料為線下維護，是否會產生同步或錯誤風險？

**目前不確定的地方：**

* 白名單驗證點是在 OAuth callback 前還是後？
* 若驗證失敗，session 是否仍被建立？

---

#### 2️⃣ Brand Token 的隔離與唯一性風險

**我們在擔心什麼：**

* token 是否在不同品牌間具有足夠隔離性
* token / domain 組合錯置時，是否可能誤登入他牌
* token 產生是否可能重複或可預測

**具體關注情境：**

* A 品牌 token + B 品牌 domain
* 舊 token 是否在新 token 產生後立即失效
* 合約終止後，token 是否仍能成功驗證

---

#### 3️⃣ Session 與實際權限的不同步風險

**我們在擔心什麼：**

* token 被更新後，既有 session 是否仍有效
* 合約終止時，已登入使用者是否會被強制登出
* 使用瀏覽器返回 / 快取是否可繞過登出流程

---

### ❓ 目前需要外部釐清的關鍵問題

以下問題若未釐清，後續測試會高度猜測性：

1. 是否能取得登入與驗證的 **實際 API flow / sequence diagram**
2. 合約終止在系統中是否有「狀態」，還是僅行政流程
3. domain alias 是否被視為有效登入條件
4. token 的生成與驗證是否為單一服務，還是分散於多處
